# Portfolio Brief — sovralys-infra-lab

## One-line summary

Personal infrastructure lab demonstrating Linux host operations, KVM/QEMU virtualization, private networking, Docker/Jupyter separation, recovery procedures and incident documentation.

## Recruiter-readable positioning

This repository is evidence for DataOps, IT Production, Data Engineering and MLOps roles in regulated environments. It shows the ability to operate below the notebook layer: servers, VMs, access patterns, service isolation, failure recovery and technical documentation.

## What this repo proves

| Evidence | Why it matters for banking/data roles |
|---|---|
| Ubuntu 24.04 host setup | Linux is the default runtime for data platforms, schedulers, APIs and containers |
| KVM/QEMU Windows VM | Enterprise data platforms often integrate legacy, vendor or Windows-only workloads |
| Private Tailscale access | Administrative access should avoid unnecessary public exposure |
| Docker/Jupyter on host | Data and ML services should run close to the Linux runtime, not inside fragile desktop stacks |
| Incident post-mortems | Production teams need root-cause analysis and corrective actions |
| Backup/recovery documentation | Reliability requires recoverable systems, not disposable experiments |
| Sanitized public docs | Public engineering documentation must avoid secrets, hostnames and operational leakage |

## Interview angles

### For Application & Data Support

> I have already operated a real Linux host with private access, VM lifecycle management, service checks and incident documentation. That helps me reason about production incidents, logs, failed services and recovery steps.

### For Data Engineer / DataOps

> This repo is the infrastructure layer behind future data projects. It documents where data workloads should run, how services are isolated, and how operational decisions are recorded.

### For MLOps

> MLOps depends on reliable runtimes. This lab shows the operational basis for hosting model-serving APIs, Jupyter environments, Docker services and monitored jobs.

### For future Data Lead trajectory

> A data lead must understand the platform under the team’s pipelines. This repo documents hands-on exposure to host operations, virtualization, private networking and failure modes.

## What this repo is not

- It is not a production cloud architecture.
- It is not a bank-grade security baseline.
- It is not a trading-performance showcase.
- It is not a substitute for the upcoming banking-data and MLOps repositories.

## Next portfolio links

This repo should be read together with future projects:

2. `fraud-mlops-control-tower` — fraud/risk ML pipeline with MLflow, Docker, CI/CD and model governance.
3. `jedha-rncp35288-portfolio` — six-block certification evidence.
